⚠️ Prüfungsfalle
TCP kann kein Multicast! Deshalb nutzt KNXnet/IP Routing zwingend UDP. Tunneling (Einzelverbindung zur ETS) nutzt TCP. Das ist ein Klassiker!
Der 3-Way-Handshake (TCP)
TCP baut eine Verbindung in exakt 3 Schritten auf:
Client → Server : SYN (Ich möchte verbinden)
Server → Client : SYN-ACK (OK, ich habe empfangen, bestätige)
Client → Server : ACK (Bestätigung empfangen — Verbindung steht!)
💡 Praxis-Tipp
Merkhilfe: TCP = Einschreiben mit Rückschein. UDP = Flugblatt in den Briefkasten. TCP bekommt Bestätigung, UDP nicht.

📦 Datenkapselung & Sockets
Encapsulation — Wie ein Paket entsteht
Wenn eine Anwendung Daten sendet, wird jede OSI-Schicht einen Header hinzufügen. Das nennt sich Encapsulation (Kapselung). Beim Empfänger wird der Prozess rückwärts durchgeführt (Decapsulation).
SchichtWas wird hinzugefügt?Größe
Layer 7 (Application)Modbus-Funktionscode / MQTT-Payload1–2 Byte (Nutzlast)
Layer 4 (Transport)TCP-Header: Ports, Sequenznummer, Prüfsumme+20 Byte
Layer 3 (Network)IP-Header: Quell-IP + Ziel-IP+20 Byte
Layer 2 (Data Link)Ethernet-Header: MAC-Adressen + Frame-Typ + FCS+14–18 Byte
GESAMT—≥66 Byte für 2 Byte Nutzlast!
⚠️ Overhead-Berechnung (Klausurrelevant!)
Payload: 2 Byte | Overhead: 64 Byte | Gesamt: 66 Byte
Payload-Anteil = (2 / 66) × 100 = 3,03%
Das bedeutet: 97% des Pakets sind Verwaltungsaufwand. Nur 3% sind echte Nutzdaten!
Overhead-Formel
Payload-Anteil (%) = (Payload / Gesamtgröße) × 100
Gesamtgröße = Payload + Overhead
Was ist ein Socket?
Eine IP-Adresse allein reicht nicht, um eine Verbindung eindeutig zu identifizieren. Ein Server kann mehrere Dienste auf derselben IP anbieten — der Port unterscheidet sie.
Socket = IP + Port + Protokoll
Beispiel Modbus: 192.168.1.50:502/TCP
Beispiel MQTT: 192.168.1.50:1883/TCP
Beispiel KNXnet/IP: 224.0.23.12:3671/UDP (Multicast!)

🌐 Subnetting & CIDR-Notation
Die Subnetzmaske teilt eine IP-Adresse in zwei Teile: den Netzwerkanteil (gleich für alle Geräte im selben Netz) und den Hostanteil (eindeutig für jedes Gerät).
Beispiel: 192.168.10.50 /24
BegriffAdresseErklärung
IP-Adresse192.168.10.50Das Gerät selbst
Subnetzmaske255.255.255.0 (/24)Erste 3 Blöcke = Netz, letzter Block = Gerät
Netzwerkadresse192.168.10.0Letzter Block = 0. Nicht vergebar!
Broadcast-Adresse192.168.10.255Letzter Block = 255. Sendet an ALLE Geräte.
Hostbereich192.168.10.1–254254 mögliche Geräteadressen
GatewayMeist .1 (z.B. .10.1)Router ins nächste Netz
Die wichtigste Kommunikationsregel
SituationWas passiert?Welche Schicht?
Quell- und Ziel-IP im GLEICHEN SubnetzDirekte Kommunikation über Switch (MAC-Adresse)Layer 2
Quell- und Ziel-IP in VERSCHIEDENEN SubnetzenPaket muss an Router (Gateway) — der leitet weiterLayer 3
⚠️ Prüfungsfalle: Subnetz-Erkennung
192.168.30.x und 192.168.40.x sind VERSCHIEDENE Netze! Der dritte Block (30 vs. 40) unterscheidet sich.
Ein Switch allein kann diese nicht verbinden — es braucht zwingend einen Router auf Layer 3.
CIDR-Notation und Präzision
CIDRSubnetzmaskeBedeutungHosts

/24255.255.255.0Gesamtes /24-Netz (254 Hosts)254

ℹ️ Firewall-Relevanz von CIDR
In ACL-Regeln: 10.0.20.50/32 = genau dieses eine Gerät (z.B. Middleware)
10.0.20.0/24 = alle 254 Geräte im IoT-Netz
Ausnahmen müssen mit /32 spezifisch über allgemeinen Regeln stehen!
Routing-Ablauf: Server A → Zähler B
Server A: 192.168.30.15/24 | Zähler B: 192.168.40.200/24 | Gateway: 192.168.30.1
Schritt 1: Server A vergleicht Netzanteil: 192.168.30.x ≠ 192.168.40.x → VERSCHIEDENE Netze
Schritt 2: Server A sendet Paket an Gateway (192.168.30.1)
Schritt 3: Router empfängt und leitet weiter ins Netz 192.168.40.x
Schritt 4: Zähler B empfängt das Paket

→ Switch ALLEINE: NEIN (kennt nur MAC-Adressen im selben Netz)
→ Router nötig: JA (Layer 3)
IP-Konflikt: Doppelte IP-Adresse
Wenn zwei Geräte dieselbe IP haben (z.B. Smart-TV und Middleware beide auf 192.168.1.50), nutzt der Switch ARP (Layer 2) zur MAC-Auflösung. Beide Geräte antworten auf ARP-Requests, der Switch speichert abwechselnd die MAC-Adresse des TV oder der Middleware → Verbindung bricht zyklisch ab ("Flapping").
📡 MQTT — Publish / Subscribe
MQTT ist das Standard-Protokoll für IoT und Gebäudeautomation. Es ersetzt das ineffiziente Polling (ständiges Fragen) durch ein Event-basiertes System: Geräte senden nur, wenn sich etwas ändert.
Die drei Rollen in MQTT
RolleBeispielAufgabe
Publisher (Sender)Sensor, Zähler, WallboxVeröffentlicht Daten unter einem Topic
Broker (Vermittler)Mosquitto, Home AssistantEmpfängt alle Daten und verteilt an Abonnenten
Subscriber (Empfänger)Aktor, Visualisierung, AppHat Topics abonniert, empfängt neue Werte automatisch
Topic-Struktur
Topics sind hierarchische Datenpfade — von allgemein nach spezifisch:
gebaeude / etage / raum / gerät / datenpunkt

Beispiele:
gebaeude/kg/technikraum/waermepumpe/vorlauf
gebaeude/carport/wallbox_01/status
gebaeude/carport/wallbox_01/ladeleistung/soll
gebaeude/carport/wallbox_01/energie/zaehlerstand
Quality of Service (QoS)
QoSNameWie?Wann nutzen?
0At most onceEinmal senden, keine Bestätigung (wie UDP)Temperaturanzeige — ein verlorener Wert ist egal
1At least onceSenden mit PUBACK-Bestätigung, kann doppelt kommenSchaltstatus — muss ankommen, Duplikat tolerierbar
2Exactly once4-Wege-Handshake — genau einmal verarbeitetZählerstände, Abrechnungsdaten
⚠️ Prüfungsfalle: QoS für Zählerstände
Zählerstände (kWh) MÜSSEN QoS 2 haben! Bei QoS 1 könnte ein Wert doppelt gezählt werden (zu hohe Rechnung). Bei QoS 0 könnte ein Wert verloren gehen (zu niedrige Rechnung). Nur QoS 2 garantiert genau einmalige Verarbeitung.
Das Retain-Flag
Normalerweise bekommt ein neuer Subscriber erst Daten, wenn der Publisher das nächste Mal sendet. Mit Retain = TRUE speichert der Broker den letzten Wert und schickt ihn sofort beim Abonnieren.
💡 Merksatz
Retain = TRUE ist wie eine Anzeigetafel: Die letzte Info bleibt stehen, auch wenn niemand gerade sendet. Neu-Ankommende sehen sofort den aktuellen Stand.
MQTT vs. Polling
EigenschaftPolling (Modbus)MQTT (Event-basiert)
NetzwerklastHoch (konstant)Gering (nur bei Änderung)
LatenzBis zu 1 AbfrageintervallFast Echtzeit
SkalierbarkeitSchlecht (120 Geräte × 1min = viele Pakete)Gut (N:M-Kommunikation)
ProtokollTCPTCP (QoS 1/2) oder UDP-ähnlich (QoS 0)

2. Firewall-Regeln (ACL) — Top-Down-Prinzip
⚠️ Kritische Regel
Die Firewall liest streng von oben nach unten. Die erste passende Regel wird ausgeführt, alle anderen ignoriert!
→ Ausnahmen MÜSSEN über den allgemeinen Regeln stehen!

ℹ️ Stateful Inspection
Erlaubt Regel 2 dem PC den Zugriff auf die Middleware, lässt die Firewall die Antwort der Middleware automatisch durch. Es wird keine explizite Rückrichtungsregel benötigt!
3. Port-Forwarding (DNAT) — Das Risiko
Botnetz-Scan-Dauer
t = Anzahl_Ports / Scan-Geschwindigkeit
65.536 Ports ÷ 300 Ports/Sek = 218 Sekunden ≈ 3,6 Minuten
⚠️ Kritisches Sicherheitsrisiko
Modbus TCP (Port 502) hat KEINE Passwort-Authentifizierung! Wer den Port findet, hat sofortigen Schreibzugriff — kann Relais schalten, Heizung verstellen usw.
→ NIEMALS Modbus TCP per Port-Forwarding ins Internet freigeben!
4. VLAN Hopping
Wenn ein unmanaged Switch an einen Trunk-Port angeschlossen wird, werden alle VLAN-Tags weitergeleitet. Alle angeschlossenen Geräte befinden sich dann in allen VLANs gleichzeitig und können auf jedes Netz zugreifen — massive Sicherheitslücke!
🔐 TLS, Zertifikate & VPN
1. HTTPS / TLS — Verschlüsselung
HTTPS verschlüsselt Daten asymmetrisch (Public/Private Key). Niemand im Netzwerk kann mitlesen — auch ein Gast im selben WLAN nicht.
⚠️ Der Browser-Fehler "Verbindung nicht sicher"
Wenn der Kunde https://10.0.20.50:8123 aufruft und eine rote Warnung sieht:

❌ Der Datenverkehr ist TROTZDEM verschlüsselt! (Self-Signed Zertifikat)
✅ Was fehlt: Die Identitätsbestätigung durch eine Certificate Authority.

Ein offizielles Zertifikat (Let's Encrypt, DigiCert) kann nur auf einen registrierten Domainnamen ausgestellt werden, NIEMALS auf eine private IP-Adresse wie 192.168.x.x oder 10.x.x.x!
ZertifikatstypVerschlüsselung?Identität bestätigt?Browser-Warnung?
Offizielles Zertifikat (CA)JaJa (für Domains)Nein
Self-SignedJa (100%)NeinJa (rote Warnung)
Kein HTTPS (HTTP)NeinNeinWarnung (unsicher)
2. VPN — WireGuard Architektur
Ein VPN kapselt Daten sicher ein und ermöglicht Fernzugriff ohne Port-Forwarding für Einzeldienste.
WireGuard-Ablauf (Smartphone → Middleware):

1. Smartphone (5G): App erzeugt Paket → Ziel-IP: 10.0.20.50 (intern)
2. Internet:        VPN verschlüsselt → neues UDP-Paket an 85.214.10.20:51820 (WAN-IP)
3. Router:         Prüft WireGuard-Schlüssel → packt aus → sieht interne Ziel-IP
4. LAN:            Routet Paket ins VLAN 20 → Middleware 10.0.20.50 empfängt
💡 Vorteil VPN gegenüber Port-Forwarding
VPN öffnet nur EINEN Port (UDP 51820) mit Schlüsselauthentifizierung. Kein direktes "Loch" in die interne Anlage. Protokolle ohne Auth (Modbus) bleiben geschützt.
3. Cloud-Verbindung ohne Port-Forwarding
Wie kann eine Cloud die Middleware steuern, ohne dass ein Port geöffnet ist?
🔑 Schlüsselkonzept: Ausgehende Verbindung
Die Middleware baut selbst eine ausgehende TCP/TLS-Verbindung zum Cloud-Server auf (Port 443 HTTPS). Diese wird durch die Stateful Inspection Firewall als established connection erkannt.

Die Firewall erlaubt den Rückverkehr des Cloud-Servers automatisch, weil die Verbindung von innen initiiert wurde.
→ Kein eingehender Port muss geöffnet werden. Der Cloud-Server gibt Befehle über diese bestehende Verbindung zurück.
⚙️ TCA-Modell & Boolesche Logik
1. Das TCA-Modell
BlockNameWas ist es?Wichtig!
TTRIGGER (Auslöser)Das Ereignis, das die Regel aufwecktFast immer eine FLANKE (Zustandswechsel)!
CCONDITION (Bedingung)Prüft, ob Regel ausgeführt werden darfImmer ein statischer ZUSTAND (TRUE/FALSE)!
AACTION (Aktion)Der Befehl, der ausgeführt wirdNur wenn T feuert UND C = TRUE
⚠️ Der klassische Flanken-Fehler (Prüfungsklassiker!)
Problem: Trigger = "Regen beginnt" (Flanke: FALSE→TRUE). Condition: Fenster ist offen.
Fehler: Um 15:00 Uhr öffnet der Kunde das Fenster manuell bei weiter regnendem Wetter.
Warum kein Eingriff? Der Trigger (Regenbeginn) feuert nur EIN MAL bei der Flanke — er feuert nicht erneut, weil der Regen nicht erneut beginnt! Das System "schläft".

Lösung: Trigger auf Zustand ändern: "SOLANGE Regen UND Fenster offen → zufahren" oder zweiter Trigger: "Fenster öffnet sich → wenn Regen → zufahren"

5. Hysterese und Feedback-Loops
Ein klassischer Fehler ist das Flapping (Schwingen):
⚠️ Feedback-Loop Problem
Regel: WENN Überschuss > 1500W → Heizstab AN
Heizstab startet → zieht 2000W → Überschuss sinkt von 3000W auf 1000W → Regel <1500W → AUS → Überschuss steigt wieder → AN → Schwingen im Sekundentakt!
💡 Lösung: Hysterese (Totband)
Falsch: EIN >1500W, AUS <1500W (selber Wert)
Richtig: EIN >3000W, AUS <1000W (Totband von 2000W)

Hysterese-Wert wählen: Mindestens so groß wie die Leistung des geschalteten Verbrauchers (hier 2000W).
6. Beispiel TCA-Design: Jalousie
Kundenwunsch: "Jalousie auf 50%, wenn Sonne stark (PV >4kW),
 nur im Sommer (Juni–August), nur wenn Fernseher AN."

TRIGGER:   PV-Leistung wechselt auf >4kW (FLANKE)
CONDITION: Monat in [Juni, Juli, August]
           AND Fernseher_Status == AN
ACTION:    Jalousie_Wohnzimmer → Position 50%
⚡ Energiemanagement, Schieflast & Peak Shaving
1. Schieflastverordnung VDE-AR-N 4100
⚠️ Gesetzliche Grenze
Die maximale Phasendifferenz (Schieflast) am Netzanschlusspunkt darf 20 A nicht überschreiten!
Einspeisung = negativer Bezugsstrom.
Symmetrische 3-phasige Verbraucher (z.B. 3-phasige Wallbox) verändern die Schieflast NICHT!
Schieflast-Berechnung
Schieflast = max(Phase) − min(Phase)
[Einspeisung zählt als negativer Wert!]

Beispiel: L1 = -16A, L2 = +6A, L3 = +2A
max = +6A, min = -16A → Schieflast = 6 − (−16) = 22A → Normverstoß!
Schieflast-Paradoxon: Wallbox auf L1 bei Einspeisung
Ausgangssituation: L1 = -16A (speist ein), L2 = +6A, L3 = +2A → Schieflast = 22A
Wallbox startet 1-phasig auf L1 mit 10A:
Neu L1 = -16A + 10A = -6A (weniger Einspeisung)
Neu L2 = +6A (unverändert)
Neu L3 = +2A (unverändert)
Neue Schieflast = +6A − (−6A) = 12A ✓ (unter 20A, OK!)

→ Die Wallbox startet REDUZIERT die Schieflast! Der Kunde irrt sich.
💡 Merksatz Schieflast
Eine Wallbox auf L1 bei starker L1-Einspeisung ist GÜNSTIG für die Schieflast! Die Einspeisung und der Bezug gleichen sich aus.
2. Batterie, C-Rate & Wechselrichter-Flaschenhals
Formeln
Kapazität (kWh) = Nennspannung (V) × Kapazität (Ah) ÷ 1000

Max. DC-Leistung (Batterie) = Kapazität (kWh) × C-Rate

Nutzbare AC-Leistung = MIN(DC-Leistung, Wechselrichter-Limit)
ParameterFormelBeispiel (48V, 300Ah, 0.5C, WR 4.6kW)
KapazitätU × Ah ÷ 100048V × 300Ah ÷ 1000 = 14,4 kWh
DC-Leistung (C-Rate)kWh × C14,4 kWh × 0,5 = 7,2 kW
Nutzbare AC-LeistungMIN(DC, WR)MIN(7,2kW, 4,6kW) = 4,6 kW
Flaschenhals—Der Wechselrichter (4,6 kW < 7,2 kW)
⚠️ Prüfungsfalle: Flaschenhals immer benennen!
Die nutzbare Systemleistung ist immer das MINIMUM aus chemischer Batterieleistung (C-Rate) und elektrischer Wechselrichterleistung. Wer ist kleiner? Das ist der Flaschenhals!
3. Peak Shaving (15-Minuten-Mittelwert)
Peak Shaving Formel
Ziel-Arbeit = Limit (kW) × 15 Minuten
Bisherige Arbeit = Leistung × vergangene Zeit
P_rest = (Ziel_Arbeit − Bisherige_Arbeit) ÷ Verbleibende_Zeit
Beispiel (Aufgabe aus dem Unterricht):
Limit: 200 kW × 15 min = 3000 kWmin (Ziel-Arbeit)
11 min bei 240 kW:    240 × 11 = 2640 kWmin (verbraucht)
Restzeit: 15 − 11 = 4 Minuten
P_rest = (3000 − 2640) / 4 = 360 / 4 = 90 kW (maximal)
Minus Grundlast (50 kW): 90 − 50 = 40 kW (für Kältemaschine)
Probeklausur-Beispiel (300/200 kW, 5+5+5 min):
Ziel: 250 kW × 15 min = 3750 kWmin
Min 0–5: 300 kW × 5 min = 1500 kWmin
Min 5–10: 200 kW × 5 min = 1000 kWmin
Bisherige Arbeit: 2500 kWmin
Restzeit: 5 Minuten
P_rest = (3750 − 2500) / 5 = 1250 / 5 = 250 kW
🔢 Modbus TCP & Endianness
1. Modbus-Grundlagen
Modbus TCP ist ein einfaches Master-Slave-Protokoll auf Layer 7, das über TCP/IP auf Port 502 kommuniziert. Kein Passwort, keine Authentifizierung.
BegriffBedeutung
Register16-Bit Speicherzelle (INT16, max. 65.535)
INT3232-Bit Wert aus 2 Registern (High Word + Low Word)
High WordDas "große" Register — der Multiplikator (× 65.536)
Low WordDas "kleine" Register — der Rest
Coil1-Bit Register (Schaltzustand)
Holding RegisterLese-/Schreibregister (z.B. 40100)
2. Endianness — Big Endian vs. Little Endian
INT32 Berechnungsformel
Gesamtwert = (High_Word × 65.536) + Low_Word

Big Endian: Register1 = High Word, Register2 = Low Word
Little Endian: Register1 = Low Word, Register2 = High Word
⚠️ Endianness-Fehler (Klausur-Klassiker!)
Register 40100: 500 | Register 40101: 5

❌ Falsch (Big Endian konfiguriert, aber Gerät sendet Little Endian):
500 × 65.536 + 5 = 32.768.005 Wh (unmöglich!)

✅ Richtig (Little Endian: Reg1 = Low Word):
5 × 65.536 + 500 = 328.180 Wh (= 328 kWh, plausibel!)
Der Azubi hat "Big Endian" konfiguriert, aber das Gerät sendet "Little Endian" (Word Swap). Register 40100 ist tatsächlich das Low Word, Register 40101 das High Word.
3. Warum Polling problematisch ist
120 Raumtemperaturregler werden jede Minute abgefragt:
Pro Paket: 2 Byte Nutzdaten + 64 Byte Overhead = 66 Byte Gesamtgröße
Payload-Anteil: 2/66 × 100 = 3,03%
120 Regler × 1/Minute = 120 TCP-Verbindungen / Minute
MQTT mit Retain würde diese Last auf Near-Zero reduzieren!

Beleuchtung (Not-Aus), Heizung (Grundwärme), Fenstersteuerung und Alarmfunktionen dürfen NIEMALS ausschließlich über Middleware realisiert werden! Die Middleware ist ein "Single Point of Failure". KNX und andere Feldbussysteme müssen eigenständig funktionieren.

Aufgabe: Overhead-Berechnung (Modbus Polling)

Rechnen
Eine Visualisierung ruft per Modbus TCP (Polling) minütlich die Werte von 120 Raumtemperaturreglern ab.
Nutzdaten: 2 Byte | Overhead: 64 Byte
a) Berechnen Sie den prozentualen Payload-Anteil.
b) Warum TCP für Modbus, UDP für KNXnet/IP Routing?
a) Payload-Anteil:
Payload / Gesamt × 100 = 2 / (2+64) × 100 = 2/66 × 100 = 3,03% Nur 3% des Pakets sind echte Nutzdaten! 97% sind Header/Overhead. Das zeigt, wie ineffizient Polling mit kleinen Datenpunkten ist.

b) TCP vs. UDP Begründung:
Modbus TCP verwendet TCP, weil Messwerte und Steuerbefehle garantiert ankommen müssen — ein verlorener Wert könnte falsche Schaltaktionen auslösen. Zuverlässigkeit > Geschwindigkeit.

KNXnet/IP Routing verwendet UDP, weil: 1. Multicast-Übertragung an alle KNX-Geräte gleichzeitig nötig ist 2. TCP kann kein Multicast (nur Unicast) 3. Geschwindigkeit wichtiger als Einzelbestätigung

Aufgabe: Schieflast-Berechnung (Wallbox)

Rechnen
Ausgangswerte am Netzanschlusspunkt:
L1 = 8A (Bezug), L2 = -5A (Einspeisung), L3 = -8A (Einspeisung)
a) Wallbox startet 1-phasig auf L1 mit 6A. Neue Ströme? Schieflast? Freigabe?
b) Wallbox 3-phasig mit 16A/Phase. Neue Ströme? Schieflast?
Aktuelle Schieflast (ohne Wallbox):
max = +8A, min = -8A → Schieflast = 8 - (-8) = 16A ✓ (unter 20A OK)

a) 1-phasig auf L1 mit 6A:
Neu L1 = 8 + 6 = 14A | L2 = -5A | L3 = -8A
Neue Schieflast = 14 - (-8) = 22A → ÜBERSCHREITET 20A! ❌ Freigabe NICHT erlaubt bei 6A auf L1!

b) 3-phasig mit 16A/Phase:
Neu L1 = 8 + 16 = 24A | Neu L2 = -5 + 16 = 11A | Neu L3 = -8 + 16 = 8A
Neue Schieflast = 24 - 8 = 16A ✓ (unter 20A, OK!) ✅ 3-phasige Ladung darf freigegeben werden!

Erkenntnis: Symmetrische 3-phasige Verbraucher verändern die Schieflast nicht (addieren auf alle drei Phasen gleich). Die Schieflast bleibt konstant.

Aufgabe: Modbus Endianness-Fehler

Rechnen
Register 40100: 500 | Register 40101: 5
Dashboard zeigt: 32.768.005 Wh
a) Wie kommt die Middleware auf diesen Wert? (Big Endian konfiguriert)
b) Tatsächlicher Wert? Welcher Endianness-Fehler?
a) Fehler-Reproduktion (Big Endian falsch):
Big Endian: Register 40100 = High Word (Multiplikator), 40101 = Low Word
500 × 65.536 + 5 = 32.768.000 + 5 = 32.768.005 Wh Das ist fast 33 MWh — völlig unrealistisch für einen Haushalt!

b) Fehlerkorrektur (Little Endian / Word Swap):
Das Gerät sendet Little Endian: Register 40100 = Low Word, 40101 = High Word
5 × 65.536 + 500 = 327.680 + 500 = 328.180 Wh = 328 kWh ✅ Das ist ein realistischer Zählerstand!

Lösung: In der Middleware-Konfiguration "Word Swap" / "Little Endian" einstellen.

Aufgabe: Flaschenhals-Logik (Wärmepumpe Boost)

Rechnen + TCA
PV: 3 kW | Hausverbrauch (ohne WP): 1 kW
Batterie: 20 kWh, C-Rate 0.5, SoC 100%
Wechselrichter max. AC: 4,6 kW
Wärmepumpe Status 4 (Boost): braucht 10 kW
Darf die Middleware den Boost freigeben?
1. Max. DC-Leistung (Batterie/Chemisch):
20 kWh × 0,5 C = 10 kW (DC, chemisch möglich) 2. Tatsächliche AC-Leistung (Flaschenhals!):
MIN(10 kW, 4,6 kW) = 4,6 kW (Wechselrichter-Limit!) 3. Verfügbare autarke Gesamtleistung:
PV-Überschuss: 3 kW - 1 kW (Hausverbrauch) = 2 kW PV-Überschuss
Gesamt: 2 kW (PV) + 4,6 kW (WR-Limit) = 6,6 kW 4. Entscheidung:
6,6 kW < 10 kW (benötigt) → ❌ NEIN, Boost NICHT freigeben!

Flaschenhals: Der Wechselrichter (max. 4,6 kW AC) — nicht die Batteriechemie (10 kW möglich).
Trotz voller Batterie kann das System nur 6,6 kW autark bereitstellen.

Aufgabe: Peak Shaving Berechnung

Rechnen
Leistungslimit: 200 kW (15-Minuten-Mittelwert)
08:00–08:11 Uhr (11 min): durchschnittlich 240 kW
IT-Grundlast: 50 kW (nicht abschaltbar)
Auf welchen Wert muss die Kältemaschine begrenzt werden?
Schritt 1: Ziel-Arbeit im Intervall
200 kW × 15 min = 3000 kWmin Schritt 2: Bisherige Arbeit
240 kW × 11 min = 2640 kWmin Schritt 3: P_rest
P_rest = (3000 − 2640) / (15 − 11) = 360 / 4 = 90 kW Schritt 4: Grundlast abziehen
90 kW − 50 kW (Grundlast) = 40 kW (für Kältemaschine) ✅ Kältemaschine darf maximal 40 kW in den restlichen 4 Minuten verbrauchen.

Rechnen

Berechne: Kapazität (kWh), Max DC-Leistung, Nutzbare AC-Leistung, Flaschenhals

1.1 — IP-Konflikt: Middleware vs. Smart-TV (4P)

4 Pkt
Middleware verliert alle 30 Sek. die Verbindung. Smart-TV hat manuell dieselbe IP 192.168.1.50. Welches Layer-2-Protokoll ist verantwortlich?
ARP (Address Resolution Protocol) — Layer 2

Beide Geräte antworten auf ARP-Broadcasts ("Wer hat 192.168.1.50?"). Der Switch-Cache wird abwechselnd mit der MAC des TV oder der Middleware überschrieben → Pakete landen zufällig am falschen Gerät → Abbruch alle ~30 Sek. (ARP-Cache-Timeout).

Lösung: IP der Middleware statisch und eindeutig vergeben.

1.2 — Wireshark: abgebrochener 3-Way-Handshake (3P)

3 Pkt
Wireshark zeigt: SYN gesendet, kein SYN-ACK empfangen. Welcher OSI-Layer?
Layer 4 — Transport (TCP)

SYN → SYN-ACK → ACK = TCP-3-Way-Handshake = Layer 4. Ursache: Firewall, geschlossener Port oder überlastete Gegenstelle.

1.3 — KNX-IP-Router nicht erreichbar, Middleware schon (3P)

3 Pkt
PC (10.0.10.5) kann Middleware (10.0.20.50) pingen, aber nicht KNX-IP-Router (10.0.20.100). Switchport leuchtet, Firewall erlaubt VLAN 10 → 20. Wahrscheinlichste Ursache auf Layer 3?
Fehlendes Standard-Gateway am KNX-IP-Router (Layer 3)

Der PC sendet korrekt. Aber der KNX-IP-Router weiß nicht, wohin er Antwortpakete an 10.0.10.5 (anderes Subnetz) schicken soll → Antwort wird verworfen.

1.4 — KNXnet/IP Routing über Layer-3-Router (3P)

3 Pkt
KNXnet/IP Routing zwischen zwei Gebäuden funktioniert nicht. Warum?
UDP-Multicast wird von Standard-IT-Routern nicht weitergeleitet

KNXnet/IP Routing nutzt Multicast-Gruppe 224.0.23.12:3671/UDP. Multicast-Routing (IGMP-Proxy/PIM) muss explizit konfiguriert werden — standardmäßig deaktiviert.
Lösung: Multicast-Routing aktivieren oder KNXnet/IP Tunneling (TCP Unicast) verwenden.

TEIL 2: Automatisierungslogik (25 Punkte)

2.1 — Außenbeleuchtung: Geht das Licht an? (5P)

5 Pkt
Trigger: PIR wechselt auf TRUE | Condition: <50 Lux | Action: Licht AN
17:00 Uhr: 60 Lux, PIR → TRUE (Licht geht nicht an). PIR bleibt TRUE. 17:15 Uhr: 40 Lux.
a) Geht Licht an? b) Wie umprogrammieren?
a) ❌ Nein!
Trigger = Flanke (einmaliger Wechsel FALSE→TRUE). Um 17:00 feuerte er einmalig. Danach bleibt PIR TRUE — keine neue Flanke, kein neuer Trigger. Das System "schläft".

b) Dual-Trigger (Best Practice): 
Trigger 1: PIR wechselt auf TRUE
Trigger 2: Helligkeit wechselt auf <50 Lux
Condition: PIR == TRUE AND Helligkeit < 50 Lux
Action:    Licht AN

2.3 — Feedback-Loop / Flapping (5P)

CONDITION = statischer Zustand (TRUE/FALSE zum Zeitpunkt des Triggers)
ACTION = Befehl, wenn Trigger feuert UND Condition TRUE

4.2 — Firewall ACL Fehleranalyse (5P)

5 Pkt
Regel 1: ALLOW 10.0.20.50/32 → WAN Port 443. Regel 2: ALLOW 10.0.20.0/24 → WAN Port 443. Regel 3: DROP ANY. Fehler und warum ist Regel 1 überflüssig?
Fehler: Regel 2 öffnet das GESAMTE IoT-Netz (254 Geräte!) für HTTPS → Kameras, Sensoren, Malware dürfen ins Internet.

Regel 1 überflüssig: Top-Down — Regel 2 trifft bereits auf 10.0.20.50/32 (Teilmenge von /24). Regel 1 wird nie als erste relevant.

Korrekt: Nur Regel 1 behalten, Regel 2 löschen.

4.4 — Self-Signed Zertifikat Browser-Warnung (4P)

4 Pkt
Browser zeigt rote Warnung bei https://10.0.20.50:8123. Kunde: "Unverschlüsselt!" — Stimmt das?
❌ Nein — der Datenverkehr ist vollständig verschlüsselt!

Self-Signed Zertifikat = Verschlüsselung 100% aktiv. Die Warnung bedeutet nur: Identität nicht durch eine Certificate Authority (CA) bestätigt. CAs stellen keine Zertifikate für private IP-Adressen aus — nur für registrierte Domainnamen.

4 Pkt
Kein DNAT geöffnet — Cloud steuert trotzdem Middleware. Wie?
Ausgehende Verbindung + Stateful Inspection:
